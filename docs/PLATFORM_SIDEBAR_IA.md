# 平台分组与经典侧边栏 IA（信息架构）草案

## 1. 目标

- 在不增加“第二套入口”的前提下，让平台相关入口在侧边栏具备可分组、可切换、可裁剪能力。
- 让“账号运营任务”和“系统运行任务”在导航语义上解耦，减少跨页面跳转成本。
- 为后续“平台布局管理器 / 平台分组切换器 / 托盘显示范围”提供统一结构约束。

## 2. 一级导航分组（固定）

- `总览`：Dashboard、公告、运行时状态。
- `账号与会话`：统一资产、会话、Wakeup/Verification。
- `连接与能力`：Provider、模型、MCP、Skills、Prompts、Tools。
- `系统与运维`：更新、日志、Report、Settings、安全能力。

说明：一级分组固定，避免“每个平台一套一级菜单”导致信息架构失控。

## 3. 平台维度（可切换）

- 在侧边栏顶部增加 `平台分组切换器`，只控制“账号与会话”分组内的内容可见性。
- 平台分组由用户定义（如 `Gemini 运营组`、`Codex 运维组`、`国内 IDE 组`）。
- 每个分组可配置：
  - 平台集合（如 `gemini/codex/cursor`）
  - 默认落地页（如资产页或会话页）
  - 是否同步到托盘快捷入口

## 4. 经典侧边栏布局规则

- 侧边栏保留“固定区 + 动态区”：
  - 固定区：一级分组与系统入口。
  - 动态区：受当前平台分组影响的二级入口（仅账号与会话域）。
- 动态区排序规则：
  - 固定优先级：统一资产 > 会话 > Wakeup。
  - 同优先级按最近访问时间排序（可关闭）。
- 动态区空态：
  - 若当前平台分组没有可见入口，显示“该分组未配置入口”，并提供“去布局管理器”快捷跳转。

## 5. 可见性策略（统一口径）

- 可见 ≠ 可操作：
  - 入口可显示，但页面内按平台能力灰显操作（避免入口抖动）。
- 分组三层作用域：
  - 导航作用域：控制入口显示。
  - 列表作用域：控制默认筛选条件。
  - 托盘作用域：控制快捷项与摘要来源。

## 6. 落地顺序建议

1. 先落地平台分组切换器（只做导航可见性）。
2. 再接入布局管理器（增删改排序 + 持久化）。
3. 最后接入托盘显示范围与跨页面默认筛选继承。

## 7. 非目标（当前阶段不做）

- 不新增独立“平台专属主页面”。
- 不改变现有页面内部的数据模型。
- 不引入平台间不同风格的侧边栏渲染器。
